Problem: The Service layer is shallow — its interface is nearly as complex as its implementation, acting entirely as a pass-through.

Solution: Apply the deletion test: remove the Service layer entirely and allow UI components to interface directly with the Repositories.

  • Delete 4 shallow wrappers.
  • Locality: logic concentrates in one module.
  • Interface shrinks; implementation absorbs wrappers.

Problem: IRoomRepository is a wide interface bundling room lifecycle, timer state, and realtime subscriptions together, making it hard to mock or understand.

Solution: Deepen the architecture by segregating the single wide interface into focused, role-specific interfaces, forming better seams.

  • Leverage: one focused interface per domain concern.
  • Tests hit one narrow interface.
  • Two adapters justify the seam (mock vs real).

Top recommendation

Collapse the shallow Service layer

This is the clearest deepening opportunity. The current Service layer is entirely shallow and provides no abstraction leverage, acting simply as a pass-through to the Repository layer. Collapsing it passes the deletion test by concentrating complexity rather than just moving it.
